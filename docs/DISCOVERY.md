# MOP v2 — Discovery / Product Brief

> **Статус:** discovery завершён, идёт проектирование (см. SPEC-PHASE-0-1.md)
> **Версия:** 0.5 (раунды 1–3 закрыты + разбор образца инвойса)
> **Дата:** 2026-07-09
> **Канонический дом документа:** репозиторий `mop` (перенесён из erp_mimirhead)

---

## 1. Контекст и цель

Студия аутсорса игрового арта, 15–40 человек, 2–3 проекта, модель оплаты **Time & Material**.
Производство живёт в Kaiten. География сотрудников: Россия, Кыргызстан, Армения, Грузия;
офис (штат) в Армении, остальные — контракт.

**Боль владельца:** процессы разрознены; после перерыва картину состояния студии приходится
собирать заново. Производительность оценивается «на глаз», нет фактуры.

**Полярная звезда:** страница-кокпит — открыл после отпуска, за 10 минут понял состояние
студии (деньги, загрузка, проекты, люди).

**Продукт НЕ делает:** не заменяет Kaiten; не является бухгалтерией (проводки, отчётность — вне контура).

---

## 2. Производственный цикл (as-is)

```
ТЗ клиента → обсуждение среза (лид+клиент+PM) → карточки в Kaiten, назначение
→ итерации ↔ фидбек клиента → приём работ, закрытие карточек
→ пул завершённых задач → внутренний апрув часов (PM готовит, Owner апрувит/корректирует)
→ апрув клиента (страница Google Sheets = один инвойс) → корректировки (PM/Owner)
→ финальный инвойс → подпись → оплата
```

Период инвойса — месяц, фактически цикл растягивается до двух.

## 4. Факты из discovery

### Пользователи и доступ
- Пользователи: **Owner, PM, возможно TeamLead**. Художники — не пользователи (пока).
- Зарплаты видит только Owner (либо разблокируется уровнем доступа).
- Нужно отключение аккаунтов сотрудников из панели.
- Системы студии: **Kaiten, Discord, Google Drive, Google Mail, SVN (будет)**.

### Маппинг Kaiten
- Space = проект; отдельный space под отпуска/прогулы. Клиентов в Kaiten нет.
- Board = субпроект (срез по локации/уровню или набор однотипных задач).
- Billable vs non-billable — отдельная доска внутри space проекта.
- Типы задач: и теги, и child-карточки с типом. Аналитика должна уметь оба среза.
- История: втянуть 1–2 года.

### Качество данных (риски)
- Часы: обычно в конце дня, некоторые — в конце недели (ретро-логирование до ~недели).
- Оценки (estimate) часто отсутствуют/не обновляются — известная проблема студии.

### Деньги
- Движения денег вносятся **вручную**: выплаты пару раз в месяц, входящие — реже раза в месяц.
  Объём — десятки транзакций в месяц. Импорт из банков не нужен.
- Категории расходов: ФОТ, налоги, софт, оборудование (разовые закупки), аренда, маркетинг,
  комиссии (в т.ч. переброска денег), обучение сотрудников, мероприятия (корпоратив, буткэмп).
- Оклады: номинированы в USD или валюте страны сотрудника; выплата в валюте страны или USD/USDT.
  У каждого индивидуально; зависит от типа занятости (штат Армения / контракт).
- Ставка клиента: сейчас одна на всех; закладываем рейт-карты (валюта, даты действия).
- Премии: ручное решение Owner по результатам проекта, года, PPR. Основание для расчёта —
  в т.ч. ежемесячные переработки. Инвойс-валюты: USD, EUR.
- Базовая валюта отчётности — **USD**. Курсы тянуть автоматически из API центробанков; USDT = USD.
- Налоги приложение **считает автоматически** (правила по стране/типу занятости).
- Историческая ставка клиента (1–2 года назад) — та же, что сейчас.

### Инвойс-цикл
- Период — месяц (фактически до двух). PM готовит, Owner финально апрувит и корректирует.
- Клиенту отдаётся ссылка на страницу Google Sheets (одна страница = один инвойс).
- Клиент режет часы **на уровне задачи**. Нужны причины-комментарии (внутренние) для
  **обеих** ступеней корректировок: внутренней (срезали сами до клиента) и клиентской.

### Посещаемость, отпуска, переработки
- Отпуска **и прогулы** — карточки с логами часов в одном отдельном Kaiten space.
- Отпуск **копится помесячно** (4 недели/год): через полгода доступно ~2 недели;
  баланс становится доступен после 6 месяцев. **Прогул сдвигает весь график отпусков вперёд**.
- Плюс 3 оплачиваемых выходных в году; их можно тратить как больничные
  («легальный прогул»: без отработки и без вычета из ЗП).
- Больничные существуют **не у всех** сотрудников, оплачиваются **частично**.
- Переработка = залогированные часы сверх нормы месяца. НЕ путать с **отработкой**
  (человек отпросился, затем вышел в выходной/праздник отработать прогул).
- Неотработанный прогул → вычет из зарплаты.
- Норма месяца — **производственный календарь страны сотрудника** (RU, KG, AM, GE).

### План/факт
- План = загрузка людей (продаём их часы): сколько времени утилизируют, сколько должны продать.
- Факт = сколько реально продали (по апруву часов в инвойсе).
- Целевая billable-утилизация: **75–80%**. План на человека, просмотр группами по команде.
- Сегодня плана нет, только факт.

### Команды
- Структура ещё не решена; идея: **мини-команды по 3–4 человека** внутри проекта,
  во главе наставник (senior / sub-lead) — «производственные единицы».
- Человек может быть на двух проектах, только если второй — внутренний проект студии
  (таких пока нет).

### Развитие сотрудников
- PPR = Personal Performance Review, раз в год.
- Дорожная карта развития сотрудника: план/факт, ставится на период (дата ОТ–ДО),
  обсуждается на 1x1.
- Графики промежуточных 1x1-созвонов — фиксировать в приложении.
- Результаты 1x1 (PM ↔ сотрудник) — фиксировать в приложении.

### Эксплуатация
- Разработка и поддержка: владелец + AI-агенты.

### Образец инвойса (страница Google Sheets, клиент Gaijin)
- Заголовок: «MimirHead to Gaijin». Одна страница = один инвойс-период, покрывает
  **три проекта одного клиента** (Enlisted, Active Matter, Line of Contact).
- Строка = **ассет** (например `american_skyscraper_building_c_middle/_bottom_a`),
  часы разбиты по **типам работ в колонках**: `ref/blockout`, `lod00`, `tex/materials`,
  `lods/cls`, `destr`, `dmg`, `set dressing`, `misc`; колонка `hours` — сумма строки.
- `specification` — текстовое перечисление сделанного (references, blockout, lod00-03,
  cls, decals, set dressing, cmp, feedback, …).
- Колонки `project` и `art manager` — выпадающие списки; `approved` — чекбокс
  **на строке-ассете**; `in payment` — чекбокс на строке.
- Внизу: строки `administration <Project>` (адм. часы биллятся по проекту,
  например Active Matter — 299 ч), `total <Project>`, общий `total` (1961 ч),
  чекбокс `total paid`.
- **Денег на листе нет — только часы**: это акт приёмки часов; финансовый инвойс
  (Word/PDF по шаблону клиента) — отдельный документ.

## 6. Фазы строительства (вертикальные срезы)

Каждая фаза — работающая польза, следующая не начинается, пока предыдущая не эксплуатируется.

**Фаза 0 — Фундамент.**
Репозиторий, CI (build+lint+тесты), auth + роли, audit log, реестр сотрудников (минимум),
надёжный синк Kaiten (транзакционный RPC, lock, retry, инкрементал по updated),
маппинг space→проект и классификация досок, производственные календари стран,
импорт истории 1–2 года.

**Фаза 1 — Billing Pipeline (главная ценность).**
Пул завершённых задач за период → внутренний апрув с корректировками и причинами →
клиентские корректировки с причинами → строки инвойса → экспорт под шаблон →
реестр инвойсов (отправлен/подписан/оплачен).

**Фаза 2 — Утилизация и аналитика v1.**
Норма часов на человека (календарь страны − отсутствия), billable %, план на человека/месяц
vs факт (апрувнутые часы), срезы: тип задач / проект / команда / клиент / сотрудник / таймлайн.
Первая версия кокпита.

**Фаза 3 — Деньги.**
Журнал движений (мультивалюта, категории, курс к базовой валюте), ФОТ из компенсаций
(+ премии, − вычеты за прогулы), P&L по месяцам и проектам (часы × рейт-карта − себестоимость).

**Фаза 4 — HR-глубина.**
Баланс отпусков (кулдаун + сдвиг за прогулы), attendance-ledger (прогул/отработка/оплачиваемые
выходные), испытательные сроки, PPR, дорожные карты развития, расписание и результаты 1x1.

**Фаза 5 — Периферия.**
Лицензии/ключи, offboarding-чеклист (Kaiten, Discord, Google, SVN, почта),
интеграция Google Sheets API, наставничество, планирование (гант/сценарии) — по готовности.

## 7. Черновик модели данных (уровень таблиц)

### `kaiten.*` — зеркало (read-only, пересинхронизируемое, без ручных полей!)
`spaces, boards, columns, cards, users, time_logs, tags, card_types` + `sync_runs` (журнал синков).

### `app.*` — мастер-данные
**Организация и люди**
- `employees` — ФИО, страна, тип занятости (staff/contractor), даты найма/испытательного/увольнения, грейд, роль, kaiten_user_id, команда
- `teams` — команда, лид
- `system_accounts` — сотрудник × система × логин × статус (offboarding)
- `licenses` — софт/ключ, кому выдан, стоимость, продление

**Клиенты и проекты**
- `clients`
- `projects` — клиент, kaiten_space_id, статус
- `board_mappings` — kaiten_board_id → проект + класс (billable / non_billable / time_off / ignore)
- `rate_cards` — клиент/проект, ставка, валюта, действует с/по

**Billing**
- `billing_periods` — проект × период × статус (draft → internal_review → client_review → approved → invoiced)
- `billing_items` — период, карточка, сотрудник, часы: raw → internal → client; причины обеих корректировок
- `invoices` — номер, проект, валюта, сумма, статусы (sent/signed/paid), даты
- `invoice_lines`

**Компенсации и деньги** (доступ: Owner)
- `compensations` — сотрудник, сумма, валюта номинала, валюта выплаты, действует с/по
- `bonuses` — сотрудник, период, сумма, основание (проект/год/PPR/переработки), комментарий
- `deductions` — вычеты (неотработанные прогулы)
- `transactions` — дата, направление, сумма, валюта, курс к базовой, категория, контрагент, проект?, сотрудник?, комментарий
- `expense_categories` — справочник (ФОТ, налоги, софт, оборудование, аренда, маркетинг, комиссии, обучение, мероприятия)

**Посещаемость и календари**
- `calendars` — страна × дата × тип дня × норма часов
- `attendance_events` — сотрудник, дата, тип (прогул/отработка/оплач. выходной/отпуск), часы, связь отработка→прогул, комментарий
- `vacation_ledger` — начисления/списания отпуска (расчётный)

**Развитие**
- `dev_roadmaps` — сотрудник, период от–до, цели (план), статус/факт
- `one_on_ones` — сотрудник, с кем, план/факт даты, результаты, связь с роадмапом
- `pprs` — сотрудник, дата, итоги, решения

**План**
- `utilization_plans` — сотрудник × месяц × плановые billable-часы (или %)

**Система**
- `audit_log` — кто, что, когда, до/после

### `analytics.*` — только views
`v_utilization`, `v_project_pnl`, `v_payroll_month`, `v_cockpit`, `v_task_type_stats`, …

## 8. Открытые вопросы

### Раунд 4 — по образцу инвойса (нужны до имплементации Фазы 1)
